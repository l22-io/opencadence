# OpenCadence - Design Specification

## Overview

OpenCadence is an open-source, self-hostable health data pipeline for developers. Thin iOS and Android companion apps collect wearable vitals (heart rate, HRV, SpO2, respiratory rate) via HealthKit and Health Connect, pushing them to a Python backend that ingests, processes, stores, and exposes data via REST and WebSocket APIs. A FHIR R4 export layer provides healthcare interoperability without forcing it on the core data model.

## Target Audience

Developers and tinkerers who want to own their wearable health data. Self-host a backend, point your devices at it, query your data via API. No vendor lock-in, no cloud dependency. Also serves as a reference architecture for anyone building health data products.

## Architecture

### Approach: Modular Monolith

Single deployable FastAPI application with five internal modules communicating through an in-process event bus. Modules have clean interfaces so they could be extracted into separate services later, but the monolith keeps self-hosting simple and contributor experience approachable.

This is a deliberate architectural choice: the internal boundaries demonstrate distributed systems thinking while the single deployment shows the judgment to avoid premature complexity.

### Modules

#### 1. Ingestion (`ingestion/`)
- Receives data from mobile apps via REST endpoints
- Validates payloads against metric registry schemas
- Normalizes timestamps and units
- Deduplicates based on (device_id, metric, timestamp)
- Emits `DataReceived` events to the internal event bus

#### 2. Processing (`processing/`)
- Subscribes to `DataReceived` events
- Runs configurable processing chains:
  - Resampling and gap filling
  - Rolling aggregations (1min, 5min, 1hr)
  - Anomaly flagging (e.g., resting HR outside personal baseline)
- Processing chains are pluggable per metric type
- Emits `DataProcessed` events

#### 3. Storage (`storage/`)
- Subscribes to `DataProcessed` events
- Writes to TimescaleDB hypertables partitioned by user and time
- Manages data retention policies (configurable, default 90 days for raw data)
- Continuous aggregates for 1-minute and 1-hour rollups (kept indefinitely)
- Handles data lifecycle

#### 4. API (`api/`)
- Query endpoints for processed data
- Filtering by time range, metric type, resolution
- WebSocket endpoint for real-time data streaming
- OpenAPI documentation auto-generated by FastAPI
- Prometheus-compatible metrics endpoint

#### 5. FHIR Export (`fhir/`)
- FHIR R4 Observation resource mapping
- Read-only export endpoint (`GET /fhir/Observation`)
- Maps internal data model to FHIR resources on the fly
- No FHIR write path - this is an export/interoperability layer

### Cross-cutting Concerns (`core/`)
- **Auth:** JWT-based. API keys for device authentication, bearer tokens for client access.
- **Event Bus:** In-process, interface-based. Designed so it can be swapped for Redis Streams or NATS later without changing module code.
- **Configuration:** Environment variables, documented in `.env.example`.
- **Logging:** Structured JSON logging.
- **Health Checks:** Readiness and liveness endpoints for K8s.
- **Metrics:** Prometheus-compatible `/metrics` endpoint.

## Data Model

### Ingest Payload Format

```json
{
  "device_id": "uuid",
  "batch": [
    {
      "metric": "heart_rate",
      "value": 72.0,
      "unit": "bpm",
      "timestamp": "2026-03-11T10:30:00Z",
      "source": "apple_watch_series_9"
    }
  ]
}
```

### Database Schema

#### `raw_samples` (hypertable)
Time-series table storing every data point as received. Partitioned by time. Retention policy configurable (default: 90 days).

| Column | Type | Description |
|--------|------|-------------|
| time | timestamptz | Sample timestamp |
| device_id | uuid | Source device |
| metric | text | Metric name (e.g., `heart_rate`) |
| value | double precision | Measured value |
| unit | text | Unit of measurement |
| source | text | Device model/source identifier |
| received_at | timestamptz | Server receive timestamp |

#### `aggregates_1min` / `aggregates_1hr` (continuous aggregates)
Automatic rollups: min, max, mean, stddev, count per device per metric. Kept indefinitely.

#### `devices`
Registered devices with API keys and metadata.

#### `anomalies`
Flagged data points with reason codes and severity.

### Metric Registry

Each supported metric type is defined in a registry:
- Name and display label
- Expected unit
- Valid range (for anomaly detection)
- Default aggregation method
- FHIR Observation mapping

Adding a new metric type = adding a registry entry + optional custom processor. This is the primary extensibility point.

### MVP Metrics

**Phase 1 (Vitals):** Heart rate, HRV, SpO2, respiratory rate, skin temperature
**Phase 2 (Activity):** Steps, workouts, calories, distance, sleep stages

## Mobile Apps

### Design Philosophy
Thin data collectors. Minimal UI. Reliable background sync. No analytics, no charts, no social features.

### iOS (Swift)
- HealthKit background delivery for all supported vitals
- Background URLSession for reliable uploads (survives app termination)
- Apple Watch data flows through HealthKit on iPhone automatically (no watchOS app needed)
- Local queue with retry logic (Core Data)
- UI: connection status, last sync time, data summary

### Android (Kotlin)
- Health Connect API for unified health data access
- WorkManager for periodic background sync
- Local queue with retry logic (Room)
- UI: connection status, last sync time, data summary

### Shared Patterns
- Payload format: JSON arrays matching the ingest format above
- Device registration: scan QR code or enter server URL + API key
- Authentication: device-level API keys issued by the backend
- No user accounts in the app
- No cloud dependency - point at any OpenCadence instance

## Deployment

### Self-host (Docker Compose)
```bash
git clone <repo>
cp .env.example .env
docker compose up
```

Three commands to a running instance. Includes:
- FastAPI backend
- TimescaleDB
- Redis
- Seed script for demo data

Resource-efficient enough to run on a Raspberry Pi.

### Production (Kubernetes)
- Helm chart with sensible defaults
- Horizontal pod autoscaler on the API
- TimescaleDB via operator or external managed PostgreSQL
- Ingress with TLS termination
- ConfigMaps and Secrets for configuration

## Security

### Application Security
- API rate limiting per device key
- Input validation at ingestion boundary
- No PII in default schema (device IDs, not user names)
- JWT authentication with key rotation support
- Encrypted at rest (database-level TDE)
- TLS in transit (enforced)

### Repository Security
- `SECURITY.md` with vulnerability disclosure process (security@email)
- Dependabot for dependency updates
- CodeQL for static analysis
- Branch protection on `main` (PR required, CI must pass, 1 review minimum, no force push)
