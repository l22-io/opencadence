# OpenCadence - Design Specification

## Overview

OpenCadence is an open-source, self-hostable health data pipeline for developers. Thin iOS and Android companion apps collect wearable vitals (heart rate, HRV, SpO2, respiratory rate) via HealthKit and Health Connect, pushing them to a Python backend that ingests, processes, stores, and exposes data via REST and WebSocket APIs. A FHIR R4 export layer provides healthcare interoperability without forcing it on the core data model.

## Target Audience

Developers and tinkerers who want to own their wearable health data. Self-host a backend, point your devices at it, query your data via API. No vendor lock-in, no cloud dependency. Also serves as a reference architecture for anyone building health data products.

## Architecture

### Approach: Modular Monolith

Single deployable FastAPI application with five internal modules communicating through an in-process event bus. Modules have clean interfaces so they could be extracted into separate services later, but the monolith keeps self-hosting simple and contributor experience approachable.

This is a deliberate architectural choice: the internal boundaries demonstrate distributed systems thinking while the single deployment shows the judgment to avoid premature complexity.

### Modules

#### 1. Ingestion (`ingestion/`)
- Receives data from mobile apps via REST endpoints
- Validates payloads against metric registry schemas
- Normalizes timestamps and units
- Deduplicates based on (device_id, metric, source, timestamp)
- Enforces batch size limits (max 1000 samples per request)
- Emits `DataReceived` events to the internal event bus
- Returns HTTP 429 when rate limit exceeded, HTTP 503 when event bus queue depth exceeds threshold

#### 2. Processing (`processing/`)
- Subscribes to `DataReceived` events
- Runs configurable processing chains:
  - Resampling and gap filling
  - Rolling aggregations (1min, 5min, 1hr)
  - Anomaly flagging (e.g., resting HR outside personal baseline)
- Processing chains are pluggable per metric type: each metric in the registry maps to an ordered list of processor class names. Contributors add a processor by subclassing `BaseProcessor` and registering it in the metric's YAML definition.
- Failed processing writes to a dead-letter table with error context for later inspection/replay
- Emits `DataProcessed` events

#### 3. Storage (`storage/`)
- Subscribes to `DataProcessed` events
- Writes to TimescaleDB hypertables partitioned by time, indexed by device_id
- Manages data retention policies (configurable, default 90 days for raw data)
- Continuous aggregates for 1-minute and 1-hour rollups (kept indefinitely)
- Handles data lifecycle

#### 4. API (`api/`)
- Query endpoints for processed data
- Filtering by time range, metric type, resolution
- WebSocket endpoint (`ws:///api/v1/stream`) for real-time streaming:
  - Clients subscribe per device and/or metric type
  - Authenticated via JWT token in the initial handshake
  - Server pushes `DataProcessed` events matching the subscription filter
  - Message format: JSON matching the query response schema
  - Client-side reconnection with last-received timestamp for gap recovery
- REST query endpoints:
  - `GET /api/v1/data?device_id=<uuid>&metric=heart_rate&start=<iso>&end=<iso>&resolution=1min`
  - Response: `{"device_id": "...", "metric": "heart_rate", "resolution": "1min", "samples": [{"time": "...", "min": 62, "max": 85, "mean": 71.3, "count": 12}]}`
  - Supports `resolution` values: `raw`, `1min`, `1hr` (maps to raw_samples or continuous aggregates)
  - `GET /api/v1/devices` - list registered devices
  - `GET /api/v1/anomalies?device_id=<uuid>&start=<iso>&end=<iso>` - query flagged anomalies
- OpenAPI documentation auto-generated by FastAPI
- Prometheus-compatible metrics endpoint

#### 5. FHIR Export (`fhir/`)
- FHIR R4 Observation resource mapping
- Read-only export endpoint (`GET /fhir/Observation`)
- Maps internal data model to FHIR resources on the fly
- No FHIR write path - this is an export/interoperability layer
- Protected by the same JWT auth as the REST API (no SMART on FHIR scopes for MVP)

### Cross-cutting Concerns (`core/`)
- **Identity Model:** No user accounts. The unit of identity is the **device**. Each device gets an API key scoped to itself. Bearer tokens (JWT) are issued for API clients (dashboards, scripts) and are scoped to a set of device IDs. A single self-hoster owns all their devices; multi-tenant isolation is out of scope for MVP.
- **Auth:** JWT-based. API keys for device authentication (write-only: ingest), bearer tokens for client access (read-only: query, FHIR export). All endpoints require authentication.
- **API Key Lifecycle:** Keys are created via a CLI command (`opencadence device register`). Keys are stored as bcrypt hashes. Keys can be revoked via CLI. No expiration by default; optional TTL configurable per key.
- **Event Bus:** In-process, interface-based (`EventBus` protocol with `publish()` and `subscribe()` methods). Default implementation uses an async queue with configurable depth limit. Can be swapped for Redis Streams or NATS by implementing the same protocol, without changing module code.
- **Error Handling:** Modules follow a fail-forward pattern. Failed event processing writes to a `dead_letter` table with full context (event payload, error, timestamp, module). A CLI command (`opencadence replay-dlq`) allows replaying failed events. Database connection failures trigger circuit breaker with exponential backoff. Ingestion returns HTTP 503 when the backend is unhealthy.
- **Configuration:** Environment variables, documented in `.env.example`.
- **Logging:** Structured JSON logging.
- **Health Checks:** Readiness and liveness endpoints for K8s.
- **Observability:** Prometheus-compatible `/metrics` endpoint. Structured logs with correlation IDs per request. Alerting and dashboarding are out of scope - users bring their own Grafana/Alertmanager.

## Data Model

### Ingest Payload Format

```json
{
  "device_id": "uuid",
  "batch": [
    {
      "metric": "heart_rate",
      "value": 72.0,
      "unit": "bpm",
      "timestamp": "2026-03-11T10:30:00Z",
      "source": "apple_watch_series_9"
    }
  ]
}
```

### Database Schema

#### `raw_samples` (hypertable)
Time-series table storing every data point as received. Partitioned by time (TimescaleDB default). `device_id` is an indexed column for fast per-device queries, not a partition dimension. Retention policy configurable (default: 90 days).

| Column | Type | Description |
|--------|------|-------------|
| time | timestamptz | Sample timestamp |
| device_id | uuid | Source device |
| metric | text | Metric name (e.g., `heart_rate`) |
| value | double precision | Measured value |
| unit | text | Unit of measurement |
| source | text | Device model/source identifier |
| received_at | timestamptz | Server receive timestamp |

#### `aggregates_1min` / `aggregates_1hr` (continuous aggregates)
Automatic rollups: min, max, mean, stddev, count per device per metric. Kept indefinitely.

#### `devices`

| Column | Type | Description |
|--------|------|-------------|
| id | uuid | Device identifier (PK) |
| name | text | User-assigned device label |
| api_key_hash | text | bcrypt hash of the API key |
| source_type | text | e.g., `apple_watch`, `pixel_watch`, `garmin_api` |
| metadata | jsonb | Arbitrary device metadata (model, OS version, etc.) |
| created_at | timestamptz | Registration timestamp |
| revoked_at | timestamptz | Null if active; set on revocation |

#### `anomalies`

| Column | Type | Description |
|--------|------|-------------|
| id | bigint | Auto-incrementing PK |
| time | timestamptz | Timestamp of the anomalous sample |
| device_id | uuid | FK to devices |
| metric | text | Metric name |
| value | double precision | The anomalous value |
| reason | text | Reason code (e.g., `out_of_range`, `spike`) |
| severity | text | `info`, `warning`, `critical` |
| context | jsonb | Additional context (threshold, baseline, etc.) |
| created_at | timestamptz | Detection timestamp |

Anomaly retention: kept indefinitely (low volume, high diagnostic value).

#### `dead_letter`

| Column | Type | Description |
|--------|------|-------------|
| id | bigint | Auto-incrementing PK |
| event_type | text | Original event type |
| payload | jsonb | Full event payload |
| error | text | Error message and traceback |
| module | text | Module that failed |
| created_at | timestamptz | Failure timestamp |
| replayed_at | timestamptz | Null until replayed |

### Metric Registry

Metrics are defined in YAML files under `backend/src/core/metrics/`. Each file defines one metric:

```yaml
# backend/src/core/metrics/heart_rate.yaml
name: heart_rate
label: Heart Rate
unit: bpm
valid_range:
  min: 20
  max: 300
aggregation: mean
processors:
  - validators.RangeValidator
  - processing.RollingAggregator
fhir:
  code: "8867-4"
  system: "http://loinc.org"
  display: "Heart rate"
```

At startup, the registry loads all YAML definitions and validates them. Adding a new metric type = adding a YAML file + optional custom `BaseProcessor` subclass. This is the primary extensibility point.

### Third-party Data Sources

The ingest API is source-agnostic. Any client that can POST the ingest payload format can feed data into OpenCadence. The mobile apps are the reference clients, but the same API supports Garmin Connect webhooks, CSV batch imports, direct BLE gateways, or custom scripts. Example: a Garmin integration would be a small adapter service that polls the Garmin API and POSTs to `/api/v1/ingest`.

### Schema Migrations

Managed via Alembic. Each migration is a versioned Python file in `backend/migrations/`. TimescaleDB-specific operations (creating/altering hypertables, continuous aggregates) are handled in migration scripts with appropriate up/down paths. Adding a new metric via the YAML registry does not require a migration - the schema is generic. Migrations are only needed for structural changes to the database.

### MVP Metrics

**Phase 1 (Vitals):** Heart rate, HRV, SpO2, respiratory rate, skin temperature
**Phase 2 (Activity):** Steps, workouts, calories, distance, sleep stages

## Mobile Apps

### Design Philosophy
Thin data collectors. Minimal UI. Reliable background sync. No analytics, no charts, no social features.

### iOS (Swift)
- HealthKit background delivery for all supported vitals
- Background URLSession for reliable uploads (survives app termination)
- Apple Watch data flows through HealthKit on iPhone when paired and powered on (no watchOS app needed). Prerequisites: iPhone must be paired with the Watch, HealthKit background delivery must be granted, iPhone must be powered on for sync to occur.
- Local queue with retry logic (Core Data). Relies on iOS Data Protection for at-rest encryption of locally queued health data.
- UI: connection status, last sync time, data summary

### Android (Kotlin)
- Health Connect API for unified health data access
- WorkManager for periodic background sync
- Local queue with retry logic (Room). Relies on Android file-based encryption for at-rest protection of locally queued health data.
- UI: connection status, last sync time, data summary

### Shared Patterns
- Payload format: JSON arrays matching the ingest format above
- Device registration: scan QR code or enter server URL + API key
- Authentication: device-level API keys issued by the backend
- No user accounts in the app
- No cloud dependency - point at any OpenCadence instance

## Deployment

### Self-host (Docker Compose)
```bash
git clone <repo>
cp .env.example .env
docker compose up
```

Three commands to a running instance. Includes:
- FastAPI backend
- TimescaleDB
- Redis
- Seed script for demo data

Designed for modest hardware. Single-user setups with a few devices can run on a Raspberry Pi 4 or equivalent.

### Production (Kubernetes)
- Helm chart with sensible defaults
- Horizontal pod autoscaler on the API
- TimescaleDB via operator or external managed PostgreSQL
- Ingress with TLS termination
- ConfigMaps and Secrets for configuration

## Security

### Application Security
- API rate limiting per device key
- Input validation at ingestion boundary
- No PII in default schema (device IDs, not user names)
- JWT authentication with key rotation support
- Encrypted at rest (database-level TDE)
- TLS in transit (enforced)

### Repository Security
- `SECURITY.md` with vulnerability disclosure process (security@email)
- Dependabot for dependency updates
- CodeQL for static analysis
- Branch protection on `main` (PR required, CI must pass, 1 review minimum, no force push)
